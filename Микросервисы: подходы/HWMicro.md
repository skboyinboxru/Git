# Домашнее задание «Микросервисы: подходы»

Вы работаете в крупной компании, которая строит систему на основе микросервисной архитектуры. Вам как DevOps-специалисту необходимо выдвинуть предложение по организации инфраструктуры для разработки и эксплуатации.

## Задача 1: Обеспечить разработку
Предложите решение для обеспечения процесса разработки: хранение исходного кода, непрерывная интеграция и непрерывная поставка. Решение может состоять из одного или нескольких программных продуктов и должно описывать способы и принципы их взаимодействия.

Решение должно соответствовать следующим требованиям:

облачная система;
система контроля версий Git;
репозиторий на каждый сервис;
запуск сборки по событию из системы контроля версий;
запуск сборки по кнопке с указанием параметров;
возможность привязать настройки к каждой сборке;
возможность создания шаблонов для различных конфигураций сборок;
возможность безопасного хранения секретных данных (пароли, ключи доступа);
несколько конфигураций для сборки из одного репозитория;
кастомные шаги при сборке;
собственные докер-образы для сборки проектов;
возможность развернуть агентов сборки на собственных серверах;
возможность параллельного запуска нескольких сборок;
возможность параллельного запуска тестов.
Обоснуйте свой выбор.

## Задача 1: CI/CD (Непрерывная интеграция и непрерывная поставка)

### Предлагаемое решение
**GitLab (Self-Managed) + GitLab Runners (на собственных серверах)**

### Компоненты решения

| Компонент | Назначение |
|-----------|------------|
| **GitLab** | Хранилище Git-репозиториев, CI/CD движок, управление проектами, безопасное хранение секретов |
| **GitLab Runners** | Агенты сборки, развернутые на собственных серверах (Docker, Kubernetes, bare-metal) |

### Соответствие требованиям

| Требование | Реализация |
|------------|------------|
| Облачная система | GitLab Self-Managed разворачивается в облаке (Yandex Cloud, AWS, GCP) либо используется GitLab SaaS |
| Система контроля версий Git | Встроенный Git-сервер |
| Репозиторий на каждый сервис | Модель "один проект (репозиторий) — один микросервис" в составе группы |
| Запуск сборки по событию из VCS | Пайплайны автоматически запускаются по push, merge request, тегам |
| Запуск сборки по кнопке с параметрами | Ручной запуск pipeline с передачей пользовательских переменных (Custom CI/CD variables) |
| Привязка настроек к каждой сборке | Переменные окружения на уровне pipeline, job, или ручной запуск с параметрами |
| Создание шаблонов для различных конфигураций | `include:` в `.gitlab-ci.yml`, централизованные шаблоны в отдельном репозитории |
| Безопасное хранение секретов | Protected/Masked variables, интеграция с HashiCorp Vault через CI/CD JWT |
| Несколько конфигураций из одного репозитория | `parallel: matrix`, `rules:`, `extends:` для разных сценариев (build, test, deploy) |
| Кастомные шаги при сборке | Произвольные команды в секции `script:` |
| Собственные Docker-образы для сборки | Поле `image:` с указанием кастомного образа; поддержка Kaniko и Docker-in-Docker |
| Развернуть агентов сборки на собственных серверах | Регистрация Runner'ов с тегами, указанием `executor = docker`, `concurrent` |
| Параллельный запуск нескольких сборок | Настройка `concurrent` в `config.toml` раннера |
| Параллельный запуск тестов | `parallel: matrix` для разделения тестов, `needs:` для параллелизации джоб |

### Обоснование выбора

GitLab предоставляет единый Application Lifecycle Management (ALM) инструмент, закрывающий потребности в хранении кода, CI/CD, реестре контейнеров и управлении секретами. Использование собственных раннеров позволяет:

- Экономить ресурсы при больших объемах сборок
- Обеспечить изоляцию и безопасность
- Использовать специфическое оборудование (GPU, высокопроизводительные CPU)

## Задача 2: Логи

### Предлагаемое решение
**ELK Stack (Elasticsearch, Logstash, Kibana) + Kafka + Filebeat**

### Компоненты решения

| Компонент | Назначение |
|-----------|------------|
| **Filebeat** | Легковесный агент на каждом хосте для сбора логов из stdout и отправки |
| **Kafka** | Буферизация и гарантированная доставка (очередь сообщений) |
| **Logstash** | Парсинг, обогащение и трансформация логов |
| **Elasticsearch** | Центральное хранилище и поисковый движок |
| **Kibana** | Веб-интерфейс для поиска, фильтрации и визуализации |

### Соответствие требованиям

| Требование | Реализация |
|------------|------------|
| Сбор логов в центральное хранилище со всех хостов | Filebeat на каждом хосте отправляет логи в Kafka → Logstash → Elasticsearch |
| Минимальные требования к приложениям (stdout) | Приложения пишут только в stdout/stderr; Filebeat подхватывает логи через драйвер Docker или чтение файлов |
| Гарантированная доставка | Kafka выступает буфером: логи сохраняются даже при недоступности Elasticsearch |
| Поиск и фильтрация по записям | Elasticsearch Query DSL, полнотекстовый поиск, фильтрация по полям (сервис, уровень, временной диапазон) |
| UI с доступом для разработчиков | Kibana с разделением по пространствам (Spaces) для разных команд |
| Ссылка на сохранённый поиск | Kibana Saved Searches — генерация прямой ссылки на конкретный запрос или дашборд |

### Обоснование выбора

ELK Stack является индустриальным стандартом для централизованного логирования. Добавление Kafka в пайплайн обеспечивает **гарантированную доставку** и защиту от пиковых нагрузок. Разработчики получают удобный интерфейс для отладки, а возможность шарить ссылки на сохраненные поиски ускоряет коммуникацию при инцидентах.

## Задача 3: Мониторинг

### Предлагаемое решение
**Prometheus + Grafana + Node Exporter + cAdvisor / kube-state-metrics**

### Компоненты решения

| Компонент | Назначение |
|-----------|------------|
| **Prometheus** | Time-series база данных, сбор метрик по модели pull |
| **Node Exporter** | Сбор системных метрик хостов (CPU, RAM, HDD, Network) |
| **cAdvisor / kube-state-metrics** | Сбор метрик контейнеров (потребление ресурсов сервисами) |
| **Grafana** | Визуализация, дашборды, агрегация данных, Alerting |

### Соответствие требованиям

| Требование | Реализация |
|------------|------------|
| Сбор метрик со всех хостов | Node Exporter на каждом хосте, Prometheus скрейпит их через service discovery |
| Метрики ресурсов хостов (CPU, RAM, HDD, Network) | Node Exporter предоставляет полный набор метрик по состоянию железа и ОС |
| Метрики потребляемых ресурсов для каждого сервиса | cAdvisor (контейнерные метрики) + kube-state-metrics (Kubernetes) или Pushgateway (для неконтейнеризованных) |
| Специфичные для сервиса метрики | Prometheus SDK в приложениях экспортирует бизнес-метрики (например, количество заказов, размер очереди) |
| UI с запросами и агрегацией | Grafana Explore с поддержкой PromQL для ad-hoc запросов и агрегаций |
| Настраиваемые панели для отслеживания | Grafana Dashboards с возможностью импорта/экспорта, версионирование дашбордов в коде |

### Обоснование выбора

Prometheus + Grafana — стандартный стек для микросервисной архитектуры, особенно в экосистеме Kubernetes. Преимущества:

- Pull-модель упрощает обнаружение сервисов и снижает нагрузку на приложения
- PromQL — мощный язык запросов для агрегации и анализа метрик
- Grafana объединяет все источники данных (Prometheus, Loki, Elasticsearch) в едином интерфейсе
- Возможность настройки алертов на основе метрик
